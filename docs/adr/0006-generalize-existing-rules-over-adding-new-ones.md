# Generalize Existing Rules Over Adding New Ones

当 Researcher agent 的方法论规则在特定场景（如提示词设计）中失效时，修复方式是泛化现有规则的作用域，而非新增规则。具体案例：提示词过度设计问题——researcher.md 已有四条控制复杂度的规则（停止扩展信号、输出边界、复杂化检测、元认知检查-过度复杂化），但它们的控制对象都是"研究过程"的复杂度，未覆盖"研究产物"的复杂度，导致提示词膨胀失控。同样，回溯规则有内容但无强制触发点，门控 B 步骤 6 只检测"沉没成本锚定"一种回溯原因，未覆盖"证据挑战框架"和"假设校准传导"。修复方式：泛化复杂化检测从"解释"到"解释+产物"，泛化门控 B 步骤 6 从单一锚定检查到三种回溯原因统一检查，升级 prompt-engineering 的 EXECUTOR AWARENESS 从前置步骤到设计约束。拒绝的替代方案：新增"复杂度门控"规则和"执行可行性假设"检查项——用加法解决加法问题，且 researcher.md 已 582 行，新增规则加剧回溯依赖。

## Considered Options

- **新增规则**：为提示词设计场景增加"复杂度门控"和"执行可行性假设"检查项。优势：规则语义精确，不改变现有规则。劣势：用加法解决加法问题；新增规则增加回溯依赖（L582 已有回溯依赖问题）；agent 倾向于"有规则就执行"，新增规则可能被机械套用产生新的过度设计。
- **泛化现有规则**：扩展现有规则的作用域覆盖新场景。优势：不增加规则数量；泛化是语义扩展而非结构变更，回溯依赖不增加；与 ADR 0001（门控是填表式协议）的设计哲学一致——门控的效力来自位置而非内容，泛化不改变位置。劣势：单条规则覆盖范围变宽，语义边界可能模糊；需要确保泛化后规则在原场景中仍然有效。

## Consequences

- 泛化后，复杂化检测和门控 B 步骤 6 的语义边界变宽。后续修改这两条规则时，需要同时考虑"研究过程"和"研究产物"两个场景，修改成本略增。
- EXECUTOR AWARENESS 从前置步骤升级为设计约束后，prompt-engineering skill 的其他规则（EXPLAIN THE WHY、STRUCTURE FOR THE READER）的执行需要先通过 EXECUTOR AWARENESS 过滤——即"先判断执行者是否需要这条信息，再决定是否写"。这改变了六条规则的执行顺序关系。
