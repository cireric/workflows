# ALGORITHM & ARCHITECTURE — 验证清单

> 本文件是 RESEARCH METHODOLOGY 在算法/架构设计场景的操作性展开。
> 触发条件：任务涉及以下任一场景时读取本文件——
> - 设计或选择算法/数据结构
> - 设计验证实验来评估算法正确性/性能/鲁棒性
> - 架构决策（模块划分、接口设计、数据流）

## 1. 问题建模验证

在 FRAME 阶段，当问题涉及算法/架构设计时，补充以下检查项：

**输入输出边界检查**：

- [ ] 输入来源：输入从哪来？是用户输入、系统生成、还是外部 API？每种来源的数据质量保证是什么？
- [ ] 输出消费者：输出被谁消费？消费方对延迟/吞吐/精度的要求是什么？
- [ ] 判断边界：算法需要做出什么判断？判断的边界是否清晰？是否存在模糊地带需要人为裁决？
- [ ] 错误后果：如果判断错了，后果是否可接受？系统有降级方案吗？降级方案的触发条件是什么？
- [ ] 前置条件：算法依赖哪些前置条件？这些条件在当前数据中是否满足？如果不满足，算法行为是什么？

**架构决策检查**：

- [ ] 当前设计决策是否先于代码存在？（如果是从代码反推的设计，标注「可能将偶然实现当成必然设计」）
- [ ] 架构约束是否真实存在？（检查「必须用 X」是因为业务需要还是历史惯性——对应门控A O2 约束真实性）

## 2. 数据理解验证

在 EXAMINE 阶段，当验证涉及算法/架构的假设时，执行以下操作序列：

**操作步骤**（按顺序执行）：

1. **读取真实数据** — 不依赖字段定义文档。使用 glob/grep/codegraph 定位数据样本，用 read 确认实际内容
2. **统计分布** — 均值、范围、缺失率、异常值。如需脚本辅助，使用 Sisyphus-Junior（输出不进入项目持久化路径）
3. **边界案例搜索** — 极端值、空值、格式不一致。使用 grep 搜索已知的异常模式
4. **数据项存在性验证** — 代码中引用的数据结构或属性，在实际样本中逐项确认。使用 codegraph 定位引用点，用 read 在数据中确认

**自检**：如果数据不支持当前设计假设 → 调整假设而非调整数据（对应系统提示词 FACTS VS ASSUMPTIONS）

## 3. 假设-实验-指标模板

在构造算法假设时，使用以下模板确保假设可验证：

```
假设：「当前规则在 [场景] 下会 [预期行为] → 因为 [因果机制] → 需要验证 [可观察指标]」
```

**假设质量检查**：

| 维度     | 好假设                                                | 差假设                       |
| -------- | ----------------------------------------------------- | ---------------------------- |
| 具体性   | 指明场景、行为、因果机制                              | 「阈值可能不对」             |
| 可验证性 | 有可观察指标，能构造反例                              | 「改一下试试」               |
| 因果性   | 解释为什么预期该行为（因果机制），而非仅描述该行为    | 「这个值应该更大」           |

**实验设计**（门控B步骤2 算法场景适配）：

| 假设类型 | 实验方法                                               | 度量定义要求                   |
| -------- | ------------------------------------------------------ | ------------------------------ |
| 正确性   | 构造已知答案的测试用例，验证算法输出与预期是否一致     | 准确率/召回率/对齐率          |
| 性能     | 在目标规模数据上测量，与基线对比                       | 延迟/吞吐/资源占用            |
| 鲁棒性   | 注入边界数据和噪声数据，观察算法是否优雅降级           | 降级触发率/降级行为正确性     |

## 4. 对比评估操作序列

**操作步骤**（按顺序执行）：

1. **建立基线** — 记录当前算法在代表性数据上的输出。使用 Sisyphus-Junior 运行基准测试（输出不进入项目持久化路径）
2. **建立 ground truth** — 如果有用户修正或已知正确答案，用它作为判断标准。如果没有，标注「缺乏 ground truth，评估基于间接指标」
3. **设计对比矩阵** — 同时跑多个版本，控制单一变量。每个版本只改一个参数
4. **量化指标** — 覆盖率、准确率、对齐率。指标必须可计算、可复现
5. **分析误分类** — 分别归类假阳性（FP）和假阴性（FN），理解根因而非仅统计数量

**证据层级映射**（算法场景特化）：

| 级别 | 算法场景证据                                |
| ---- | ------------------------------------------- |
| 1    | 算法测试用例通过                            |
| 2    | 静态分析/类型检查通过                       |
| 3    | 代码读取确认算法逻辑与设计一致              |
| 4    | 文档/注释中的算法描述                       |
| 5    | 设计文档中的预期行为描述                    |
| 6    | 未验证的算法假设                            |

## 5. 边界与降级检查清单

在验证算法/架构设计时，逐项检查：

**边界值**：
- [ ] 恰好落在阈值边界的数据，算法行为是否符合预期？
- [ ] 边界两侧的行为是否一致（避免 off-by-one）？
- [ ] 空输入/零值/负值的处理是否正确？

**跨场景**：
- [ ] 同一个算法在不同类型项目/数据中是否仍然成立？
- [ ] 算法的前提条件在目标场景中是否满足？

**假阳性**：
- [ ] 哪些场景会错误触发？触发频率的估算依据是什么？
- [ ] 错误触发的后果是否可接受？是否有人工复核机制？

**数据缺失**：
- [ ] 如果某个输入数据项不存在，算法如何降级？
- [ ] 降级路径是否在设计中显式声明，还是隐式依赖默认值？
- [ ] 降级后的输出对消费者是否可接受？

## 6. 迭代归因纪律

**约束**：

- 一轮只改一个变量。同时改多个变量无法归因。
- 用独立数据集做回归测试，防止修复 A 引入 B。
- 如果多轮迭代没有明显改善 → 回到 FRAME 重新理解问题（触发 [→FRAME] 回溯，说明触发原因：迭代收益递减，问题理解可能不完整）

**回溯锚点**：迭代中最可靠的证据锚点通常是「ground truth 测试用例」——如果存在通过的正确性测试，从该测试覆盖的行为重新出发。
