# PROMPT ENGINEERING

> 当任务涉及为其他 LLM 编写提示词时，加载本 skill。
> 抽象化原则（泛化非补丁、从案例提取规则）遵循系统提示词 ABSTRACTION FIRST，本 skill 不重复展开。

## 与系统提示词的接口

| 本 skill 规则            | 系统提示词对应概念        | 关系                             |
| ------------------------ | ------------------------- | -------------------------------- |
| EXECUTOR AWARENESS       | 无（纯增量）              | 提示词特有的维度                 |
| CONTEXT BOUNDARY         | 无（纯增量）              | 提示词特有的维度                 |

纯增量：EXECUTOR AWARENESS、CONTEXT BOUNDARY、STRUCTURE FOR THE READER。

规则冲突时：系统提示词的 ABSTRACTION FIRST 优先，本 skill 的抽象化约束作为提示词场景的操作细化。

## EXECUTOR AWARENESS

写任何提示词之前，必须先回答：

1. 谁执行这个提示词？——什么模型？上下文窗口多大？擅长什么？不擅长什么？是 API 调用还是在 Coding Agent 中的代理？
2. 执行者看到提示词时的信息状态——它知道完整的任务背景吗？还是只看到你写给它的这几句话？
3. 你与执行者的关系——你是上级在写指令、平级在协作、还是下级在汇报？

你是在为**特定的执行者**写提示词，不是为「理想的通用 LLM」写。同一个任务，给 128k 模型和 1M 模型的写法完全不同，为推理模型和非推理模型的写法也完全不同，为 Flash 级别模型和旗舰级别模型的写法也完全不同。不问执行者直接写，是大部分 prompt 翻车的根源。

执行者分析不是一次性步骤，而是提示词中每条指令的过滤条件。如果执行者分析表明执行者无法有效使用某条指令，该指令应删除或简化，而非保留并期望执行者自行忽略。每写一条指令，都对照执行者分析判断：这个执行者需要这条指令吗？能执行这条指令吗？

## CONTEXT BOUNDARY

你的上下文 ≠ 执行者的上下文。

你阅读完整对话历史后明白的事情，执行者可能不知道。你省略的「显然」信息，执行者不会自动补全。你以为的行业常识，执行者可能从未接触过。

提示词中必须显式包含执行者需要知道的一切。不能依赖「对方应该知道」的假设。

## 提示词的抽象化约束

> 遵循系统提示词 ABSTRACTION FIRST 原则。以下补充提示词场景的具体约束。

**补丁式修正的识别**：提示词中出现以下模式时，说明未完成抽象化——

- 针对某个历史错误的禁令（「不要直接调用外部 API」→ 只阻止这一次，没阻止同类错误）
- 引用具体案例的禁令（「不要学 XXX」→ 换个类似错误仍会犯）
- 时间锚定的补丁（「上个月出现过 XXX 问题」→ 新场景同样问题复现）

**正确做法**：理解行为偏差的根因（能力限制？角色认知偏差？指令模糊？），用原理性约束替代禁令。

**从案例提取规则**（对应 ABSTRACTION FIRST 的「未完成抽象化」判断）：

- 错误：用户提供条件 A+B+C 的示例 → 提示词写「如果 A 且 B 且 C 则特殊处理」→ 下游把具体组合当锚点
- 正确：从示例提取「当多个风险信号的组合超出常规模式时，需升级处理」→ 规则不依赖具体条件值

## EXPLAIN THE WHY

对于每个约束，解释它为什么重要，而不是用 ALWAYS/NEVER 强压。

有效：「输出格式必须让变量名可读，因为下游脚本按变量名匹配」

无效：「禁止使用缩写变量名」

解释 why 利用了模型的理解能力，让它能在未见场景中做出正确判断。强制指令在边界场景中很容易失效，而原理性约束不会。

这是 ABSTRACTION FIRST 在提示词中的应用——解释 why 本质上是提供规则的因果机制，使规则可在未见场景中泛化。

## STRUCTURE FOR THE READER

提示词的组织方式应该降低执行者的认知负担：

- 最重要的指令放最前面（执行者注意力依次衰减）
- 同类内容放一起（降低跳跃成本）
- 步骤式优于段落式（降低遗漏概率）
- 示例优于抽象描述（降低歧义）

## SELF-REVIEW: CONCRETENESS CHECK

每次写完提示词草稿后，必须执行一次具体性审查：

1. 搜索草稿中是否包含：项目名、角色名、具体剧情片段、文件路径
2. 如果包含：说明没有完成抽象化，需要把具体案例替换为抽象规则
3. 审查标准：这条提示词换一个完全不同的领域和场景是否仍然成立？

此审查是系统提示词 ABSTRACTION FIRST 自检（「移除具体名称后是否仍然成立且同样清晰」）在提示词场景的操作化。
