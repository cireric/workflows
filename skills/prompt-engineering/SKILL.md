# PROMPT ENGINEERING

> 当任务涉及为其他 LLM 编写提示词时，加载本 skill。

## EXECUTOR AWARENESS

写任何提示词之前，必须先回答：

1. 谁执行这个提示词？——什么模型？上下文窗口多大？擅长什么？不擅长什么？是 API 调用还是在 Coding Agent 中的代理？
2. 执行者看到提示词时的信息状态——它知道完整的任务背景吗？还是只看到你写给它的这几句话？
3. 你与执行者的关系——你是上级在写指令、平级在协作、还是下级在汇报？

你是在为**特定的执行者**写提示词，不是为「理想的通用 LLM」写。同一个任务，给 128k 模型和 1M 模型的写法完全不同，为推理模型和非推理模型的写法也完全不同，为 Flash 级别模型和旗舰级别模型的写法也完全不同。不问执行者直接写，是大部分 prompt 翻车的根源。

## CONTEXT BOUNDARY

你的上下文 ≠ 执行者的上下文。

你阅读完整对话历史后明白的事情，执行者可能不知道。你省略的「显然」信息，执行者不会自动补全。你以为的行业常识，执行者可能从未接触过。

提示词中必须显式包含执行者需要知道的一切。不能依赖「对方应该知道」的假设。

## GENERALIZE, DON'T PATCH

通过具体案例修正提示词行为会过度特化规则，因为执行者会把具体案例当锚点，在案例未覆盖的场景中失效。

错误写法：
- 「请注意，不要直接调用外部 API」（针对某个历史错误的补丁——只阻止了这一次，没阻止其他类似的错误）
- 「不要学 XXX」（依赖具体案例——换一个类似错误仍会犯）
- 「上个月出现过 XXX 问题」（无法跨场景泛化——新的场景中同样的问题会再次出现）

正确做法：
- 理解行为偏差的**根因**（是能力限制？角色认知偏差？还是指令模糊？）
- 用原理性约束替代禁令
- 解释 why 比说 what-not-to-do 更有效

判断标准：一条提示词规则如果在其他项目/其他模型中无法成立，则说明它过于特化。

## ABSTRACT FROM EXAMPLES

当你为其他 LLM 编写提示词时，你的亲身案例只应用于提取抽象规则，不得直接用作 few-shot 样本，除非你明确知道需要 few-shot。

错误做法：
- 用户提供了一个多条件分类的示例：「某条记录满足条件 A、B、C 时需特殊处理」
  → 提示词里写「如果条件=A 且条件=B 且条件=C，则特殊处理」
  → 下游 LLM 把这个具体条件组合当锚点，换了任何一个条件值的类似情况全部漏判

正确做法：
- 用户提供了一个多条件分类的示例：「某条记录满足条件 A、B、C 时需特殊处理」
  → 从中提取规则：「当多个风险信号的组合超出常规模式时，需升级处理」
  → 规则不依赖具体条件值，可以覆盖所有多条件组合场景

判断标准：如果一条提示词离开当前项目后无法正常工作，说明它过度特化了。

## EXPLAIN THE WHY

对于每个约束，解释它为什么重要，而不是用 ALWAYS/NEVER 强压。

有效：「输出格式必须让变量名可读，因为下游脚本按变量名匹配」

无效：「禁止使用缩写变量名」

解释 why 利用了模型的理解能力，让它能在未见场景中做出正确判断。强制指令在边界场景中很容易失效，而原理性约束不会。

## STRUCTURE FOR THE READER

提示词的组织方式应该降低执行者的认知负担：

- 最重要的指令放最前面（执行者注意力依次衰减）
- 同类内容放一起（降低跳跃成本）
- 步骤式优于段落式（降低遗漏概率）
- 示例优于抽象描述（降低歧义）

## SELF-REVIEW: CONCRETENESS CHECK

每次写完提示词草稿后，必须执行一次具体性审查：

1. 搜索草稿中是否包含：项目名、角色名、具体剧情片段、文件路径
2. 如果包含：说明没有完成抽象化，需要把具体案例替换为抽象规则
3. 审查标准：这条提示词换一个完全不同的领域和场景是否仍然成立？
