# ALGORITHM DESIGN

> 当任务涉及算法/架构设计时加载本 skill。

## 定位

本 skill 与系统提示词的 RESEARCH METHODOLOGY 互补而非重叠：

| 层级         | 位置                  | 职责                                                    |
| ------------ | --------------------- | ------------------------------------------------------- |
| 研究方法论   | RESEARCH METHODOLOGY  | 给出通用研究框架：多假设、验证优先、事实与假设区分        |
| 算法设计     | 本 skill 规则 1-5     | 在算法/架构设计场景中提供可操作的方法论细化               |

系统提示词给出研究框架基线，本 skill 在基线上增加算法/架构设计场景的操作细节。

## 规则 1：假设构造

**目标**：确保算法假设可验证、可证伪，而非模糊直觉。

构造算法假设时，使用以下模板：

```
假设：「当前规则在 [场景] 下会 [预期行为] → 因为 [因果机制] → 需要验证 [可观察指标]」
```

**假设质量检查表**：

| 维度     | 好假设                                                | 差假设                       |
| -------- | ----------------------------------------------------- | ---------------------------- |
| 具体性   | 指明场景、行为、因果机制                              | 「阈值可能不对」             |
| 可验证性 | 有可观察指标，能构造反例                              | 「改一下试试」               |
| 因果性   | 解释为什么预期该行为（因果机制），而非仅描述该行为    | 「这个值应该更大」           |

## 规则 2：实验设计

**目标**：为每个假设设计可复现的验证实验。

实验按验证目标分为三类：

| 假设类型 | 实验方法                                               | 度量定义要求                   |
| -------- | ------------------------------------------------------ | ------------------------------ |
| 正确性   | 构造已知答案的测试用例，验证算法输出与预期是否一致     | 准确率/召回率/对齐率          |
| 性能     | 在目标规模数据上测量，与基线对比                       | 延迟/吞吐/资源占用            |
| 鲁棒性   | 注入边界数据和噪声数据，观察算法是否优雅降级           | 降级触发率/降级行为正确性     |

**度量定义要求**：每个度量必须满足——
- 可计算：有明确的计算公式或判定标准
- 可复现：相同输入下多次运行结果一致
- 有基线：能回答「与什么相比」

## 规则 3：数据理解验证

**目标**：在依赖数据之前，先理解数据的真实分布和边界特征。

**操作步骤**（按顺序执行）：

1. **读取真实数据** — 不依赖字段定义文档。定位数据样本并确认实际内容
2. **统计分布** — 均值、范围、缺失率、异常值
3. **边界案例搜索** — 极端值、空值、格式不一致
4. **数据项存在性验证** — 算法中引用的数据结构或属性，在实际样本中逐项确认

**自检**：如果数据不支持当前设计假设 → 调整假设而非调整数据

> **适用前提**：本规则仅在任务涉及代码库数据验证时执行。纯算法设计任务跳过。

## 规则 4：对比评估

**目标**：通过受控对比获得可靠的算法选择依据。

**操作步骤**（按顺序执行）：

1. **建立基线** — 记录当前算法在代表性数据上的输出
2. **建立 ground truth** — 如果有用户修正或已知正确答案，用它作为判断标准。如果没有，标注「缺乏 ground truth，评估基于间接指标」
3. **设计对比矩阵** — 同时跑多个版本，控制单一变量。每个版本只改一个参数
4. **量化指标** — 覆盖率、准确率、对齐率。指标必须可计算、可复现
5. **分析误分类** — 分别归类假阳性（FP）和假阴性（FN），理解根因而非仅统计数量

**证据层级映射**（通用）：

| 优先级 | 证据类型              | 说明                                      |
| ------ | --------------------- | ----------------------------------------- |
| 1      | 实验验证              | 测试用例通过、基准测试复现                |
| 2      | 静态分析              | 类型检查、复杂度分析、不变量验证          |
| 3      | 代码审查              | 人工走查确认逻辑与设计一致                |
| 4      | 文档参考              | 设计文档、注释、外部文献中的算法描述      |
| 5      | 未验证假设            | 基于直觉或经验的推测，未经验证            |

> 高优先级证据可覆盖低优先级证据；同级证据冲突时需进一步调查。

## 规则 5：迭代归因

**目标**：确保每次迭代产生可学习的经验，而非盲目试错。

**约束**：

- 一轮只改一个变量。同时改多个变量无法归因
- 用独立数据集做回归测试，防止修复 A 引入 B
- 如果多轮迭代没有明显改善 → 回到问题理解阶段重新分析（触发回溯，说明触发原因：迭代收益递减，问题理解可能不完整）

**回溯锚点**：迭代中最可靠的证据锚点通常是「ground truth 测试用例」——如果存在通过的正确性测试，从该测试覆盖的行为重新出发。

## 与系统提示词的接口

| 本 skill 规则       | 系统提示词对应概念        | 关系                                                         |
| ------------------- | ------------------------- | ------------------------------------------------------------ |
| 规则1 假设构造      | MULTIPLE HYPOTHESES       | 假设构造是多假设框架在算法场景的操作细化：模板化 + 质量检查表 |
| 规则2 实验设计      | VERIFY OR DON'T CLAIM     | 实验设计是验证优先原则在算法场景的操作性扩展                   |
| 规则3 数据理解验证  | FACTS VS ASSUMPTIONS      | 数据理解验证是事实与假设区分在数据依赖场景的具体展开           |
| 规则4 对比评估      | VERIFY OR DON'T CLAIM     | 证据层级映射是验证优先原则在评估场景的具体化                 |
| 规则5 迭代归因      | MULTIPLE HYPOTHESES       | 迭代归因是多假设框架在迭代优化场景的延伸——单变量控制 + 回溯机制 |

纯增量（无系统提示词对应）：规则3 中的「适用前提」声明（算法设计与数据验证的边界划分）、规则4 中的「分析误分类」步骤（算法场景特有的根因分析要求）。

规则冲突时：系统提示词的 RESEARCH METHODOLOGY 优先，本 skill 提供算法/架构设计场景的操作细化。
