# ALGORITHM & ARCHITECTURE

> 当任务涉及算法或架构设计时，加载本 skill。

## DESIGN FROM FIRST PRINCIPLES

算法设计的起点是业务需求，不是现有代码。

设计前先回答：
- 这个算法的输入输出是什么？输入从哪来，输出被谁消费？
- 算法需要做出什么判断？判断的边界是否清晰？
- 如果判断错了，后果是否可接受？系统有降级方案吗？
- 算法依赖哪些前置条件？这些条件在当前数据中是否满足？

## KNOW YOUR DATA

在设计具体规则前，必须先理解数据分布。

- 读取真实数据，而不是依赖字段定义文档
- 统计分布：均值、范围、缺失率、异常值
- 寻找边界案例：极端值、空值、格式不一致
- 验证数据项是否存在：代码中引用的数据结构或属性，要在实际样本中逐项确认

如果数据不支持你的设计假设，调整假设而非调整数据。

## MODEL THE PROBLEM, NOT THE CODE

设计算法时，用业务语义建模，不要用当前代码的实现细节建模。

- 对业务直觉做假设：例如「某类输入的响应时间应随数据规模线性增长」
- 用数据验证假设：读取实际数据，确认分布是否匹配直觉
- 从验证结果提炼规则：例如「当数据显示某类输入的分布集中在特定区间时，判定边界应覆盖该范围」
- 把规则落地为代码：先有规则，再有代码

如果先读代码再反推业务逻辑，容易把偶然实现当成必然设计。

## HYPOTHESIS-DRIVEN EXPERIMENT

每次改动都应基于明确假设。

好假设：
  「当前规则在 X 场景下会漏掉 Y 类情况 → 因为数据分布偏了 → 需要调整判定边界」
  「某个数据项在实际样本中不存在 → 导致下游全部走默认分支 → 需要修正数据引用」

差假设：
  「阈值可能不对」（太模糊，无法验证）
  「改一下试试」（没有预期结果，无法判断成败）

## COMPARATIVE EVALUATION

算法的效果必须可衡量、可对比：

1. 建立基线 — 记录改动前的输出
2. 建立 ground truth — 如果有用户修正，用它作为判断标准
3. 设计对比矩阵 — 同时跑多个版本，控制变量
4. 量化指标 — 覆盖率、准确率、对齐率
5. 分析 FP 和 FN — 分别归类，理解根因

## EDGE CASE ANALYSIS

任何阈值或规则改动必须主动寻找边界：

- 边界值：恰好落在阈值边界的数据会怎样？
- 跨场景：同一个规则在不同类型项目中是否成立？
- 假阳性：哪些场景会错误触发？后果是否可接受？
- 数据缺失：如果某个输入数据项不存在，系统如何降级？

## ITERATE, DON'T OVERFIT

- 一轮只改一个变量。同时改多个变量无法归因。
- 用独立数据集做回归测试，防止修复 A 引入 B。
- 如果多轮迭代没有明显改善，回到第一步重新理解问题。
