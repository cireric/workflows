# SOURCE CRITIQUE

> 当任务涉及 web 信息源收集、评估或引用时，加载本 skill。

## 定位

本 skill 与 MISSION 的信息源体系互补而非重叠：

| 层级       | 位置                    | 职责                                                  |
| ---------- | ----------------------- | ----------------------------------------------------- |
| 信息源分级 | MISSION WORLD KNOWLEDGE | 给出 tier 基线："这类来源的一般可信度是什么"          |
| 降级检测   | 本 skill 规则 1-3       | 来源类型区分、利益冲突识别、6步溯源流程、证据强度评估 |
| 偏差校正   | 本 skill 规则 4         | 识别信息收集中的系统性偏差并主动校正                  |

MISSION 的信息源分级给出 tier 基线，本 skill 在基线上增加同 tier 内的来源质量评估和偏差校正。

## 规则 1：利益冲突识别

**目标**：检测信息源是否存在未声明的商业利益，导致内容偏离客观基线。

利益冲突的检测方法因来源类型不同而异——对评测文章查赞助声明有效，对行业报告查赞助声明则无效（应查商业模式）。以下按来源类型给出各自的识别信号和检查方法：

| 来源类型       | 利益冲突信号                                                                                           | 检查方法                                                                              |
| -------------- | ------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------- |
| 评测/对比文章  | 被评测方赞助或提供样品；含购买链接/优惠码；只列优点不列缺点                                            | 搜索文中是否含"赞助""合作""广告"声明；检查是否有独立购买证据                          |
| 行业报告       | 报告发布方是被分析行业的参与者；结论有利于发布方商业利益                                               | 检查发布方商业模式；对比独立来源的结论                                                |
| 专家意见       | 专家在该领域有商业利益；专家雇主与结论相关                                                             | 检查专家机构隶属和利益声明                                                            |
| 社区讨论       | 发帖者可能是品牌代理人；内容过于统一/模板化                                                            | 检查发帖历史；对比多个独立讨论                                                        |
| 系统性生态偏差 | 评测者推荐前提嵌入特定生态系统/地域/平台；推荐逻辑基于"你已在此生态"而非跨生态对比；不提及替代生态选项 | 标注评测的地理/生态前提；搜索目标市场本地来源；检查推荐结论是否在评测者前提外仍然成立 |

**降级规则**：发现利益冲突信号时，将该来源可信度降低一个 tier。无法确认也无否认时，标注"利益冲突未确认"，引用时使用限定语。

**系统性生态偏差说明**：前4种来源类型的利益冲突是**可归属的**——某个具体主体有具体利益。系统性生态偏差是**不可归属的**——评测者没有做错任何事，但其受众/地域/平台生态使推荐系统性偏向某个方向。这不是评测者的利益冲突，而是推荐前提不可迁移（与规则4的"前提不可迁移"偏差联动）。

## 规则 2：数据溯源

**目标**：确保引用的主张可追溯到一手来源，标注无法溯源的主张。

**操作流程**（6步，不可跳过）：

1. 遇到引用数据/结论 → 检查是否标注了一手来源
2. 已标注 → 验证一手来源存在且内容与引用一致
3. 未标注 → 尝试搜索定位一手来源
4. 无法定位 → 标注 UNVERIFIED，该主张不得作为高确信度结论的支撑
5. 一手来源存在但内容与引用不一致 → 标注"引用失真"，以一手来源为准
6. 被评价对象可能已变更 → 检查对象是否存在未声明的版本变更（硬件改版、软件更新、策略调整、内容修订）；对比不同时期评测结论；标注评测对应的对象版本

**引用链断裂**：A 引用 B 引用 C，但 B 的引用无法定位 C → 在 A 处标注"引用链断裂于 B→C"，降低 A 权重。

**对象变更说明**：步骤6处理的情况与前5步不同——来源本身可能完全准确，引用链也完整，但**被评价的对象在评测后发生了未声明的变更**。适用于硬件改版（型号名不变但组件替换）、软件更新（API 行为变更但端点不变）、策略调整（服务条款修改但未通知用户）、内容修订（文档静默更新但 URL 不变）等场景。此时不是"引用失真"，而是"源准确但对象已变"。

## 规则 3：证据强度评估

**目标**：评估具体证据的统计意义，区分观察性陈述与因果性结论。

| 维度       | 强证据                | 弱证据                  | 降级条件                                                                     |
| ---------- | --------------------- | ----------------------- | ---------------------------------------------------------------------------- |
| 样本量     | N ≥ 统计显著性要求    | N = 1 或未报告          | N=1 用户体验 → 降级为"个案观察"                                              |
| 样本代表性 | 随机抽样 / 全量数据   | 自选样本 / 便利样本     | 自选样本 → 标注"选择偏差风险"                                                |
| 对照组     | 有对照组 / 有基线对比 | 无对照 / 仅"使用后"描述 | 无对照 → 将"X有效"降级为"使用X后观察到Y"                                     |
| 可复现性   | 独立来源复现          | 仅单一来源              | 多源看似独立但实为同源改写（如AI内容农场）→ 标注"伪独立验证"，降级为单一来源 |

**核心原则**：单个案例不构成证据。任何"X 好用/有效/推荐"的结论，必须回答"与什么相比"——缺乏基线对比时，将结论降级为观察性陈述。

**伪独立验证说明**：AI改写和内容农场使"多源验证"的可靠性下降。多个网站发表相似结论看似独立验证，实际可能是同一来源的改写。验证独立性的方法：(1) 追溯最早发表源；(2) 检查各站是否有独立测试证据（一手数据、原创截图、独家发现）；(3) 无独立测试证据的后续来源标注为"非独立来源"。

## 规则 4：偏差校正

**目标**：识别信息收集过程中的系统性偏差，主动寻找校正信号。

### 4.1 信息收集偏差

| 偏差类型   | 表现                       | 校正方法                                            |
| ---------- | -------------------------- | --------------------------------------------------- |
| 幸存者偏差 | 只看到成功案例             | 搜索负面评价、退货率、投诉数据                      |
| 确认偏差   | 只收集支持预设结论的证据   | 刻意搜索反例（与 MISSION MULTIPLE HYPOTHESES 协同） |
| 选择性报告 | 只报告有利数据             | 寻找完整数据集或独立验证                            |
| 锚定效应   | 第一个找到的信息成为判断锚 | 先收集多源信息再形成判断，避免过早收敛              |

前4种偏差是**信息收集过程**中的偏差——收集了错误或不全的信息。校正方向是找更多、更全面的信息。

### 4.2 推理前提偏差

| 偏差类型     | 表现                                           | 校正方法                                                                             |
| ------------ | ---------------------------------------------- | ------------------------------------------------------------------------------------ |
| 前提不可迁移 | 结论在评测者前提内正确，但前提不适用于目标情境 | 显式化评测者的隐含前提（地域、生态、法规、可用性）；检查前提是否可迁移到当前研究情境 |

前提不可迁移是**推理前提**层面的偏差——信息可能完全正确，但结论依赖的隐含前提（地理前提、技术栈前提、法规前提、可用性前提等）在目标情境中不成立。校正方法不是找更多信息，而是**显式化并检验前提**。

**常见的不可迁移前提**：

| 前提类型   | 识别信号                                                                                   | 检验方法                                                                                       |
| ---------- | ------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------- |
| 地域前提   | 评测者所在市场与目标市场不同；推荐基于本地可得性/生态/法规；产品在目标市场不可用或体验不同 | 标注评测的地理前提；搜索目标市场本地来源；检查产品在目标市场的可用性和支持；对比多地域独立来源 |
| 生态前提   | 评测者默认受众处于某生态系统；推荐逻辑基于"你已在此生态"而非跨生态对比                     | 检查推荐结论是否在评测者生态外仍然成立；搜索替代生态中的评测                                   |
| 法规前提   | 结论依赖特定法规环境（如 GDPR、FDA 审批、特定行业标准）                                    | 标注法规前提；检查目标情境的法规是否兼容                                                       |
| 可用性前提 | 推荐的产品/服务在目标市场不可用或体验不同；价格/服务/支持在目标市场不同                    | 检查目标市场的实际可用性和定价                                                                 |

> 注：地域前提和生态前提的详细识别信号和校正方法，见 `@./references/bias-patterns.md` 中"跨市场适用性"和"英文科技评测媒体"章节。

### 4.3 评论区校正

主内容存在偏差风险时，评论区/讨论区可提供反例、事实性纠错、利益冲突揭露。但评论区本身也有偏差（积极参与者偏向极端意见），需平衡使用。

## 与系统提示词的接口

| 本 skill 规则  | 系统提示词对应概念         | 关系                                                           |
| -------------- | -------------------------- | -------------------------------------------------------------- |
| 规则1 利益冲突 | WORLD KNOWLEDGE 信息源分级 | 在 tier 基线上增加同 tier 内的利益冲突检测                     |
| 规则2 数据溯源 | VERIFY OR DON'T CLAIM      | 数据溯源是 VERIFY OR DON'T CLAIM 在 web 信息源场景的操作性扩展 |
| 规则3 证据强度 | VERIFY OR DON'T CLAIM      | 证据强度评估是证据层级在 web 信息源场景的具体展开              |
| 规则4 偏差校正 | MULTIPLE HYPOTHESES        | 偏差校正（来源层面）与假设层面确认偏差协同——双轨校正           |

纯增量（无系统提示词对应）：规则4.2 推理前提偏差（前提不可迁移是独立维度）、规则2步骤6 对象变更（web 信息源特有的版本追踪需求）。

规则冲突时：系统提示词的 RESEARCH METHODOLOGY 优先，本 skill 提供 web 信息源场景的操作细化。
